## AncientChinese 项目 · AI 辅助开发规则（CLAUDE.md）

本文件约定在本项目中使用 AI 助手（包括但不限于 Claude / ChatGPT 等）时的基本规则，保证团队协作一致性、代码质量与可追溯性。

---

## 1. 使用范围与原则

- AI 主要用于：
  - 代码草稿、重构建议与单元测试样例
  - 文档初稿（接口说明、使用示例、注释等）
  - Bug 排查思路、性能与安全性建议
  - 前后端接口对齐、数据结构设计讨论
- AI 生成内容**必须经过人工审阅与修改**后才能提交到仓库。
- 对业务逻辑、数据安全、权限控制等敏感部分，必须由人类开发者做最终决策。

---

## 2. 代码风格与技术栈约束

- 前端：
  - 技术栈：原生 HTML / CSS / JavaScript
  - 不随意引入新的前端框架或 UI 库，如需引入须经过团队讨论。
  - 组件交互与页面跳转要与现有 Figma 原型保持一致。
- 后端：
  - 技术栈：Python + FastAPI
  - API 设计应优先参考 [docs/api.md](./docs/api.md) 与现有接口风格。
  - 路由、请求体、响应体命名需与前端约定保持统一。
- 数据库：
  - 使用 MySQL，表结构变更必须同步更新相关文档（如 docs/database.md）。
- 通用：
  - 不在代码中硬编码密钥、账号、Token 等敏感信息。
  - 优先复用已有工具函数、封装组件与接口，而不是在多个地方重复造轮子。

---

## 3. 分支与提交规范（AI 参与时）

- 分支策略：
  - 功能开发优先在 `feature/*` 分支上进行，例如：`feature/zhangxianwen-integration`。
  - 与 `develop` 的同步通过 `git fetch origin` + `git merge origin/develop` 或 `git rebase origin/develop` 完成。
- 提交信息（commit message）建议采用简洁英文前缀：
  - `feat:` 新功能
  - `fix:` 修复 Bug
  - `refactor:` 重构（无功能变化）
  - `docs:` 文档相关
  - `chore:` 其他杂项
- 当提交中包含 AI 生成或大量参考 AI 的内容时，建议在提交信息正文中标注：
  - 例如：`Co-authored-by: AI assistant (reviewed by <your-name>)`

---

## 4. 文档与规范优先级

- 设计与说明类文档（示例）：
  - 项目说明：[README.md](./README.md)
  - 前端说明：[docs/frontend.md](./docs/frontend.md)
  - 后端说明：[docs/backend.md](./docs/backend.md)
  - 架构设计：[docs/architecture.md](./docs/architecture.md)
  - 数据库设计：[docs/database.md](./docs/database.md)
  - API 设计：[docs/api.md](./docs/api.md)
  - 个人贡献说明：`docs/contributions/**`
- AI 在生成涉及以上文档内容的方案时，应**优先对齐现有文档**，不要擅自改变已约定的：
  - 接口路径、请求 / 响应结构
  - 页面结构与导航流程
  - 术语、命名风格与数据流向

---

## 5. 前后端协作与接口约定

- 使用 AI 辅助设计 / 修改接口时：
  - 必须同步更新前端与后端对应部分（例如接口路径、字段名）。
  - 如接口变更影响他人（后端 / 前端 / 架构），需要在 PR 或说明文档中明确标注。
- 若 AI 给出的接口设计与现有 `docs/api.md` 不一致：
  - 以当前文档与已实现代码为准。
  - 如需要升级接口，由人类开发者主导设计和评审。

---

## 6. 安全性与数据隐私

- 向 AI 提问时，避免直接粘贴以下敏感内容：
  - 数据库账号密码、云服务密钥、第三方平台 Token
  - 含有真实隐私数据的样本（如真实用户数据）
- 如需构造示例数据，优先使用脱敏 / 虚构数据。
- 任何疑似泄露敏感信息的内容不得提交到仓库。

---

## 7. 使用 AI 进行重构 / 大改动的流程

当使用 AI 对某个模块进行较大改动（例如：重构前端页面路由、重写某个 FastAPI 接口）时，建议遵循以下流程：

1. 在 issue 或 TODO 中写清楚重构目标与边界。
2. 让 AI 给出 **方案与示例代码**，而不是直接全量替换。
3. 人工筛选、拆分任务，逐步实现，并在每个阶段进行：
   - 本地运行与手动验证
   - 尽量补充 / 修正测试（如有测试框架）
4. 在 PR 描述中说明哪些部分由 AI 辅助生成，便于代码审查时重点关注。

---

## 8. 冲突解决与历史操作

- 使用 AI 参与 Git 操作（例如：合并分支、回退版本）时，应遵守：
  - 重要操作（如 `git reset --hard`、`git push --force`）必须明确说明影响范围。
  - 在执行前确认当前分支、远程跟踪分支以及是否有未提交改动。
  - 尽量通过合并（merge）而不是暴力覆盖，除非团队达成共识。
- 如果因 AI 建议导致历史记录混乱或分支状态异常，应：
  - 立即记录当时的命令与状态
  - 与团队成员沟通后，选择合适的恢复方案（reflog / revert / reset）。

---

## 9. 何时不要依赖 AI

- 需求不清晰、产品方向未定时，不依赖 AI 直接「拍脑袋」生成实现。
- 涉及课程评分 / 个人考核的核心思考题（尤其是需要独立完成的部分），应以**个人理解**为主，AI 只作查缺补漏。
- 团队已就某个技术方案达成一致时，不随意让 AI 推翻已有决策，除非有明确、更优的理由并经过讨论。

---

## 10. 本文件的维护

- 当团队工作方式或技术栈发生明显变化时，应同步更新本文件。
- 修改 CLAUDE.md 时，建议使用提交前缀：`docs: update CLAUDE rules`。
- 如对 AI 使用方式存在分歧，可先在 issue 中讨论，再更新到本文件中。
